20191019 ⟪為遺留代碼寫單元測試⟫ 上課筆記

觀念

  • 不要需求沒講完就開始寫，不要需求講完沒確認就開始寫
  • 一次只測試一件事，不一定是指一個 assertion，看定義
  • 不要用 if、for、switch、try catch，測試裡面不應該再有邏輯
  • test name 應該是敘述情境
  • 測試可以反應主程式是否有壞味道，是否有高耦合…
  • unit test 要先有 CI 不然沒有價值
  • data-driven test： 用情境切分測試，不要用 data provider 把所有情境都合併在一起

data provider
優點：快速
缺點：因為 code 不會 duplicate 所以不會重構測試程式，當情境越來越多，data provider case 越多，若這時候還是綠燈 ，就會累積很多可能是不同情境的測試，導致測試程式不易理解與修改，接著測試就開始變髒了，就沒有人繼續寫測試了

如何導入

理想：
不寫測試就不 code review，不 review 就不上線
現實：

  • 透過一點 over-design 穩固 code 和 test，例如強制寫 interface 然後禁止用 new
  • code review 前先看測試，修 ticket 要附帶測試，code coverage 只能往上不能下降

constructor 若是需要注入的參數很多 可能是違反單一職責原則

Assertion package 的好處

  • 可以比較物件
  • 可以比較集合
  • 表達力增加
  • 節省時間
  • 減少錯誤可能

測試的表述方式

寫測試是說明情境，而不是在測 function，需求的關鍵字應該出現在裡面，並且其餘的都不應該出現

做法：

把共用的 function 抽出去，並用其組出情境
常用的 function 命名

  • given
  • when
  • then
  • should

案例一： Isolated test

找到依賴， extract 它，然後建立一個 SUT 測試子類 override 依賴函式，此為手動做出假物件來隔絕依賴
(參考 extract & override 單元測試的藝術 ch3、ch4)

解法：

SUT

steps

  • 找出依賴
  • extract method of dependency
  • private -> protected

SUT 測試子類

steps

  • 建立一個 SUT 測試子類
  • ovrerride 有 dependency 的 function
  • 決定子類的值

限制

static & 不能繼承的地方不能用

副作用：

被抽出的 production code 測試不會測到(因為測試會跑到 SUT 子類的 override function) -> code coverage 會掉

案例二： 依賴注入 ＆ 依賴反轉 (IOC = DI)

SUT 重構 steps

  • 抽出依賴成 field
  • 建立 interface
  • 初始化放到 constructor
  • 外部注入依賴

修改好的好處：

  1. production 可以從高耦合變成低耦合，並且舊有的無參數和新的有參數寫法都不會有問題
  2. 可以開始測試

重點：
把依賴交由外部決定，SUT 專注業務邏輯，這樣測試才會變得可行

補充：
做假的粒越小，測試程式越好理解，所以如果做假整個工廠，程式會必較不好讀

案例二改用 mock lib 實作

  • mock 背後也是 new instance ， 等同於手刻 fake class
  • production code 除了 DTO、factory，都不要 new ，一律做DI，而且用 interface

案例三： 需求異動:登入失敗紀錄 log

單元測試三種方式

  • 驗回傳值
  • 驗狀態
    老師打學生，學生被打不開心（狀態改變），但不說（沒有回傳值）
  • 驗相依物件互動
    目標 function 有沒有被呼叫

注意測試壞味道

  • 過度指定（測試太敏感不穩固）

mock 驗證可以不驗到太細 例如 log message 只要驗到有沒有互動
但是如果有遇到 bug 可以再加上去

Spy and Mock

Spy

  • spy 為後置驗證，做完才驗證，接近 3A 原則
  • 只驗證在乎的事情， 其他沒驗的東西無所謂
  • 測試比較沒那麼敏感，必較不容易壞

Mock

  • act 前先下定義，先設定白名單
  • 如果有任何跟定義不同的情況就會噴錯
  • 比spy 嚴格的驗證方式

note:
php spy 沒有 should not receive 的寫法

使用 makePartial

就像是經過一個 wrapper (可以看成 decorator)

測試方向

developer:輔助開發

QA:測邊界值以及各種情況

變異測試

production code 有任何異動，測試都應該不過， 如果沒有紅燈代表測試不夠完整

Code coverage

  • production code 95% 可能都不是主要核心 -> 無法貢獻 coverage -> 數字不好看
  • 5% 核心才是重要的，有測試價值的

coverage report 可以看出差異：刺激思考

  • 可能是 test case 不足
  • 可能是 dead code

結論：不要拿 code coverage 當 KPI

沒事不要把舊 code 重構以及寫測試

  • 因為改完後沒有資源人力和各種情境去測試改完的 code，所以是危險的

  • 適合的改動時間為 fix 和異動時加上測試，這樣才會有各種 code review、 pair programming 、 e2e test 、 manual test、、UAT test、回歸測試以及其他測試的保護

  • 不要開重構 task 異動時才一起做重構和加上測試

寫測試是買保險，買保險是需要付出代價的
應該投資在最有價值是的，例如

  • $、人命
  • 維護成本很高、很常壞掉的東西 -> 每次有需求異動都會改到而且很容易改錯的程式
  • 常用的功能、共用元件
  • bug 、ticket、issue

相對趨勢 > 絕對數字 -> 從今天開始只為自己現在開始寫的 code 寫測試 -> coverage 會慢慢上升

三層式設計（onion 架構）

UI <-> business service <-> data

core <-> adapter <-> 3rd party service
unittest 整合測試

production 實作
戳 adapter 看跟第三方是否有通

案例：
googel login 不通
production 登入不通三分鐘內跟 3rd party 通知

e2e
ec: 加到購物車、結帳
博弈：下注

stored procedure & repository

  • 當成黑箱測試
  • 直接 test db
  • test 裡面做 arrange & assert 都是 ORM去做
  • 只要測試前 arrange，然後驗證修改完的結果 

如何測試 legacy 嚴重的 code

先確認需求和情境寫出一個 test case -> 紅燈 -> 找到依賴 切掉它 -> 反覆直到綠燈

重要：不要先看程式碼，只要看方法簽章就好了，這樣只會先為核心代碼寫測試

  • CRUD 要不要測試？
